# Sena Agent - 스스로 발전하는 에이전트 프레임워크 모노레포

# 스펙 중심 개발

모든 패키지는 하위에 ./specs 폴더를 가지며, 각 패키지가 해결하고자 하는 문제와 역할, 작동 방식, 제약사항과 인터페이스(GUI, API, CLI 등), 구현 전략 등을 포함한다.
모든 작업은 스펙을 중심으로 이루어지며, 모든 코드 수정사항은 스펙을 수정한 뒤 이루어져야 한다.

## 스펙 폴더 구조

모든 스펙은 각 패키지의 `specs/` 폴더에 위치한다.

```
packages/<group>/<package>/specs/
  index.md               # 상위 스펙 (패키지의 목표, 요구사항, 범위)
  <responsibility>.md    # 상세 스펙 (책임 단위별)
```

예시:

```
packages/core/specs/
  index.md               # @sena-ai/core 상위 스펙
  agent.md               # Agent 책임 단위
  orchestrator.md        # Orchestrator 책임 단위
  turn-engine.md         # TurnEngine 책임 단위
  ...

packages/integrations/slack/connector/specs/
  index.md               # slack-connector 상위 스펙
  events.md              # 이벤트 처리
  verify.md              # 요청 검증
  ...
```

## 개발 플로우

스펙 중심 개발은 **새로운 기능 개발**과 **기존 기능 수정** 두 가지 플로우로 나뉜다. 두 플로우 모두 "스펙 먼저, 코드 나중" 원칙을 따르지만, 기존 기능 수정은 변경의 영향 범위를 분석하는 계획 단계가 추가된다.

---

### 플로우 1: 새로운 기능 개발

```
스펙 작성 → (사용자 검토/수정) → 스펙 Freeze → 구현
```

새로운 기능은 기존 스펙이 없으므로 처음부터 스펙을 작성한다.

#### 1-1. 스펙 작성

아래 "스펙 작성법" 섹션의 절차(1단계 → 2단계 → 3단계)를 순서대로 따른다.

1. **니즈 정렬**: 사용자 요청에서 Problem과 Success Criteria를 추출한다.
2. **상위 스펙 작성**: `specs/index.md`를 작성한다. 목표, 요구사항, 수용 기준, 범위 경계를 정의한다.
3. **상세 스펙 작성**: 책임 단위별로 `<responsibility>.md`를 작성한다. 4관점(A/B/C/D) 체크리스트를 순서대로 채운다.

**핵심 규칙**:

- 상위 스펙이 합의되기 전에 상세 스펙으로 넘어가지 않는다.
- 상세 스펙 파일 간 의존관계가 정리되기 전에 구현으로 넘어가지 않는다.
- 불확실한 부분은 가정(Assumption)과 미해결 질문(Open Question)으로 명시한다.

#### 1-2. 스펙 검토 및 수정

사용자에게 작성된 스펙을 제시하고 피드백을 받는다.

- **검토 대상**: 상위 스펙 → 상세 스펙 순서로 검토한다.
- **수정 반영**: 피드백을 스펙에 반영하고, 변경된 부분의 영향을 받는 다른 스펙 항목도 함께 갱신한다.
- **재합의**: Stable 항목이 변경되면 반드시 사용자와 재합의한다.
- **Freeze**: 사용자가 최종 승인하면 스펙을 Freeze한다.

**반복 조건**: 사용자가 승인할 때까지 검토 → 수정을 반복한다. 각 반복에서 변경 사유를 기록한다.

#### 1-3. 구현

Freeze된 스펙을 기반으로 코드를 작성한다.

- 스펙의 AC(수용 기준)를 테스트로 먼저 변환한다.
- 상세 스펙의 Realization 섹션에 정의된 전략을 따른다.
- 구현 중 스펙과 충돌이 발견되면, 코드를 임의로 변경하지 말고 스펙 수정 절차(플로우 2)를 따른다.

---

### 플로우 2: 기존 기능 수정

```
스펙 수정 계획 → (사용자 검토/수정) → 스펙 수정 → 스펙 Freeze
  → 구현 수정 계획 → (사용자 검토/수정) → 구현
```

기존 기능 수정은 이미 Freeze된 스펙이 존재하므로, 변경의 영향 범위를 먼저 분석하고 계획을 세운 뒤 스펙과 구현을 순차적으로 수정한다.

#### 2-1. 스펙 수정 계획 작성

기존 스펙을 분석하여 수정이 필요한 범위와 영향을 파악한다.

수정 계획에 포함할 내용:

| 항목                 | 설명                                                          |
| -------------------- | ------------------------------------------------------------- |
| **변경 동기**        | 왜 기존 스펙을 수정해야 하는가 (버그, 요구사항 변경, 개선 등) |
| **영향 분석**        | 변경이 영향을 미치는 스펙 파일 목록과 각 파일의 변경 범위     |
| **의존성 파급**      | 변경되는 스펙에 의존하는 다른 스펙/모듈의 파급 효과           |
| **안정성 분류 확인** | 변경 대상이 Stable/Flexible/Experimental 중 어디에 해당하는지 |
| **변경 항목 목록**   | 수정할 FR/NFR/AC ID와 각각의 변경 내용 요약                   |
| **리스크**           | 변경으로 인한 잠재적 위험과 완화책                            |

**핵심 규칙**:

- Stable 항목 변경은 반드시 사용자 재합의가 필요하다.
- Forward Dependency(상위 스펙에 없는 계약에 대한 의존)가 생기지 않도록 주의한다.
- 변경 범위가 넓으면 단계적 수정 전략을 계획에 포함한다.

#### 2-2. 스펙 수정 계획 검토 및 수정

사용자에게 수정 계획을 제시하고 피드백을 받는다.

- 영향 분석의 누락 여부를 확인한다.
- 변경 범위가 적절한지 (너무 넓거나 좁지 않은지) 합의한다.
- 리스크와 완화책이 충분한지 검토한다.
- 사용자가 승인할 때까지 계획을 수정한다.

#### 2-3. 스펙 수정

승인된 계획에 따라 스펙 파일을 수정한다.

- 변경 대상 파일만 수정한다. 계획에 없는 파일을 임의로 변경하지 않는다.
- 각 수정에 변경 사유를 기록한다.
- 파일 간 Traceability(상위 스펙 ↔ 상세 스펙 연결)를 유지한다.
- 수정된 스펙은 사용자 승인 후 다시 Freeze한다.

#### 2-4. 구현 수정 계획 작성

수정된 스펙을 기반으로 코드 변경 계획을 작성한다.

구현 수정 계획에 포함할 내용:

| 항목               | 설명                                                                   |
| ------------------ | ---------------------------------------------------------------------- |
| **변경 파일 목록** | 수정이 필요한 소스 파일과 각 파일의 변경 내용 요약 (소스코드는 최소화) |
| **변경 순서**      | 파일 간 의존관계를 고려한 수정 순서                                    |
| **테스트 전략**    | 변경에 따른 테스트 추가/수정/삭제 계획                                 |
| **마이그레이션**   | DB 스키마, API 버전, 설정 등 마이그레이션이 필요한 경우의 전략         |
| **롤백 계획**      | 문제 발생 시 이전 상태로 복원하는 방법                                 |

#### 2-5. 구현 수정 계획 검토 및 수정

사용자에게 구현 수정 계획을 제시하고 피드백을 받는다.

- 변경 파일 목록의 누락 여부를 확인한다.
- 변경 순서가 의존관계를 올바르게 반영하는지 검토한다.
- 테스트 전략이 수정된 AC를 충분히 커버하는지 확인한다.
- 사용자가 승인할 때까지 계획을 수정한다.

#### 2-6. 구현

승인된 구현 수정 계획에 따라 코드를 수정한다.

- 계획된 순서대로 파일을 수정한다.
- 각 변경 단위마다 관련 테스트를 실행하여 회귀를 확인한다.
- 구현 중 계획에 없는 변경이 필요하면, 추가 계획을 작성하고 사용자 승인을 받는다.
- 모든 AC가 통과하는지 최종 검증한다.

| 상황                                                | 적용 플로우                                     |
| --------------------------------------------------- | ----------------------------------------------- |
| 새 패키지 또는 새 기능을 처음부터 만드는 경우       | 플로우 1: 새로운 기능 개발                      |
| 기존 스펙이 없는 레거시 코드에 기능을 추가하는 경우 | 플로우 1: 새로운 기능 개발 (스펙을 새로 작성)   |

| 버그 수정으로 스펙의 AC와 동작이 달라지는 경우      | 플로우 2: 기존 기능 수정                        |
| 버그 수정이지만 스펙의 AC 범위 내인 경우            | 스펙 수정 없이 구현만 수정 (계획 → 검토 → 구현) |
| 리팩토링으로 외부 동작이 바뀌지 않는 경우           | 스펙 수정 없이 구현만 수정 (계획 → 검토 → 구현) |

## 스펙 작성법

### 원칙

- **기능이 아니라 문제를 먼저 정의하라.** "무엇을 만드는가"가 아니라 "무엇이 달라져야 성공인가"를 먼저 합의한다.
- **코드 전에 스펙을 쓴다.** 모든 코드 수정은 스펙 수정 이후에 이루어진다.
- **불확실한 것은 숨기지 않는다.** 가정(Assumption)과 미해결 질문(Open Question)으로 분리한다.
- **Freeze는 무단변경 금지이지, 변경 금지가 아니다.** 변경 시 사유와 영향 범위를 기록하고 사용자와 재정렬한다.

### 절차 요약

```
1. 니즈 정렬 → 2. 상위 스펙 → (사용자 합의/Freeze) → 3. 상세 스펙 → (Freeze) → 구현
```

각 단계는 이전 단계가 합의되기 전까지 다음 단계로 넘어가지 않는다.

---

### 1단계: 니즈 정렬

사용자 요청에서 다음 두 가지를 추출한다:

| 항목                 | 질문                                                |
| -------------------- | --------------------------------------------------- |
| **Problem**          | 현재 무엇이 문제인가? (배경이 중요하면 여기에 포함) |
| **Success Criteria** | 무엇이 달라지면 성공인가?                           |

정렬 과정에서 반복하는 작업:

- 더 나은 해결책이 있으면 **스펙 옵션**(효과/비용/리스크/트레이드오프 포함)으로 제안
- 현황 조사: As-Is 동작, 제약, 인터페이스/계약, 의존성, 실패 케이스
- 요구사항 **Commitment Level** 분류: **Committed**(이번에 반드시) / **Planned**(추후 반드시) / **Candidate**(할 수도) / **Never**(절대 안 함)
- 수용 기준(AC)은 Given/When/Then으로 검증 가능하게

---

### 2단계: 상위 스펙

패키지의 `specs/index.md`로 작성한다.
소스코드나 구현 세부사항을 넣지 않는다. 목표와 범위만 명확히 한다.

```markdown
# [프로젝트명]

## 한 줄 요약

"누가 / 무엇을 / 왜 / 어떤 상태로 만든다"

## 문제 정의

- 현 상태(As-Is), Pain Point (배경·우선순위 근거가 중요하면 여기에 포함)

## 목표 & 성공 지표

- 정량/정성 기준, Definition of Done

## 용어 정의

## 요구사항

- FR-001: ... [Committed/Planned/Candidate/Never]
- NFR-001: ... (성능, 보안, 비용 등 — 사용자 합의 필수)

## 수용 기준 (AC)

- AC-001: Given ... / When ... / Then ... (관련: FR-001)

## 범위 경계 (Non-goals)

- 명시적 금지 사항
- 이번 범위 밖 (차기 후보)

## 제약 & 가정

## 리스크 & 완화책

- 발생 가능성 / 임팩트 / 완화 / 조기 신호

## 검증 계획

- AC를 어떻게 검증하는가
```

**안정성 분류**: 각 항목에 **Stable**(변경 시 재합의) / **Flexible**(AC 범위 내 조정 가능) / **Experimental**(중단 기준 명시) 태그를 붙인다.

**Freeze 조건**: 상위 스펙이 합의되기 전까지 상세 스펙으로 넘어가지 않는다.

---

### 3단계: 상세 스펙

상위 스펙을 **책임 단위**로 분해하여 파일별로 작성한다. 목적은 구현자가 임의 해석 없이 작업할 수 있도록 계약과 경계를 고정하는 것이다.

#### 파일 분할 기준 (하나 이상 해당 시 분리)

- 독립 구현/테스트/배포 가능한 **책임 경계**
- 별도 상태 전이(State Machine)를 갖는 **상태 경계**
- 별도 API/CLI/화면을 갖는 **인터페이스 경계**
- 보안·결제 등 **고위험 영역**

#### 4관점 체크리스트 (A/B/C/D)

모든 상세 스펙 파일은 아래 네 관점을 **순서대로** 작성한다.

| 관점               | 질문                        | 핵심                                                        |
| ------------------ | --------------------------- | ----------------------------------------------------------- |
| **A. Behavior**    | 어떤 변화가 발생하는가?     | Trigger → 상태변화 → 결과. 실패 시나리오 포함               |
| **B. Constraint**  | 절대 깨지면 안 되는 것은?   | 성능, 보안, 정합성 등. 측정/검증 방법 명시                  |
| **C. Interface**   | 외부와의 계약은?            | API/CLI/GUI 스키마. 에러 코드. 멱등성                       |
| **D. Realization** | 어떤 전략으로 만족시키는가? | 모듈 경계, 상태 모델, 동시성, 실패 처리, 배포, 마이그레이션 |

> A를 먼저 쓰고 → B로 제한하고 → C로 계약 고정 → D로 전략 설계. 순서를 어기지 않는다.

#### 상세 스펙 파일 템플릿 (약식)

```markdown
# [책임 단위명]

## 한 줄 요약

## 상위 스펙 연결 — 관련 FR/NFR/AC ID

## Behavior

- Actor / Trigger / Preconditions
- Main Flow / Alternative Flow
- Outputs / Side Effects / Failure Modes

## Constraints

- ID / 설명 / 범위 / 측정 방법 / 검증

## Interface (해당 유형만)

- API: Endpoint, Method, Request/Response Schema, Status Codes
- CLI: Command, Flags, Exit Codes
- GUI: View ID, User Actions, UI States

## Realization

- 모듈 경계, 상태 모델, 동시성, 실패 처리, 배포, 마이그레이션

## Dependencies

- Depends On / Blocks / Parallelizable With
- ⚠ 상위 스펙에 없는 계약에 의존(Forward Dependency)은 금지

## AC — Given / When / Then (엣지 케이스 + 제약 위반 포함)
```

#### 상세 스펙 Freeze 규칙

- 모든 파일이 상위 스펙과 Traceability를 갖기 전까지 Freeze하지 않는다.
- 파일 간 의존관계가 명확히 정리되기 전까지 구현으로 넘어가지 않는다.
