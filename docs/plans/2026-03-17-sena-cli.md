# @sena-ai/cli — sena CLI 패키지 구현 계획

## 목표 개요
에이전트 디렉토리에서 보일러플레이트(`start.ts`, `worker.ts`, `restart.sh`)를 제거하고, `sena` CLI 하나로 에이전트 라이프사이클을 관리한다.

## 문제 (니즈)
현재 sena-v2 에이전트는 다음 파일들이 에이전트마다 거의 동일하게 복붙된다:
- `start.ts` (29줄) — orchestrator 엔트리포인트
- `worker.ts` (10줄) — worker 엔트리포인트
- `restart.sh` (46줄) — 프로세스 재시작 스크립트

에이전트마다 고유한 것은 `sena.config.ts`와 `.env`뿐인데, 나머지가 관리 부담을 만든다.

## 솔루션 (목표)
`@sena-ai/cli` 패키지를 만들어 `sena` 명령어로 에이전트 라이프사이클을 관리한다.

에이전트 디렉토리는 이렇게 최소화된다:
```
sena.config.ts    # 유일한 설정 파일
.env              # 환경변수
.sena/            # 컨텍스트/메모리
package.json      # @sena-ai/cli 포함
```

## 비목표 — 하면 안되는 것
- 기존 `@sena-ai/core`의 orchestrator/worker 내부 로직 변경
- HTTP 기반 management API 추가 (기존 SIGUSR2/SIGTERM 시그널 방식 유지)
- 에이전트 스캐폴딩/init 기능 (향후 가능)

## 비목표 — 범위 밖
- `sena init` (프로젝트 초기화) — 향후 추가 가능
- `sena deploy` — 향후 추가 가능
- Web dashboard — 별도 프로젝트

## 확정된 주요 의사결정
1. **프로세스 관리**: PID 파일(`.sena.pid`) + OS 시그널 (현 restart.sh 방식의 정제 버전)
2. **Config 로드**: `tsx` register를 활용해 `sena.config.ts`를 동적 import
3. **포트 해석**: config의 `orchestrator.port` → `.env`의 `SENA_PORT` → 기본값 3100 순으로 fallback
4. **Worker 진입점**: CLI가 내장 worker-entry를 제공. `SENA_CONFIG_PATH` 환경변수로 config 경로 전달
5. **Arg parsing**: `commander.js` 사용
6. **데몬 모드**: `sena start`는 기본 foreground, `-d` 플래그로 백그라운드 데몬

## 상세 실행 계획

### Task 1: CLI 패키지 스캐폴딩 (의존: 없음)
- `packages/cli/` 디렉토리 생성
- `package.json` (name: `@sena-ai/cli`, bin: `{ "sena": "./dist/cli.js" }`)
- `tsconfig.json` (composite, project reference)
- 루트 `tsconfig.json`에 cli reference 추가
- `pnpm-workspace.yaml`은 이미 `packages/*` 와일드카드라 변경 불필요
- 의존성: `commander`, `dotenv`, `@sena-ai/core` (workspace:*)

### Task 2: Config Loader 구현 (의존: Task 1)
- `src/config-loader.ts`
- `.env` 로드 (dotenv)
- `sena.config.ts` 경로 탐색 (CWD 기준, --config 옵션으로 오버라이드)
- tsx register (`tsx/esm`)로 TypeScript config 동적 import
- 포트 해석 로직: config.orchestrator.port → process.env.SENA_PORT → 3100

### Task 3: PID 관리 모듈 구현 (의존: Task 1)
- `src/pid.ts`
- PID 파일 경로: CWD의 `.sena.pid`
- 함수: `writePid(pid)`, `readPid()`, `removePid()`, `isProcessAlive(pid)`
- stale PID 감지 (프로세스 존재 여부 확인)

### Task 4: Worker Entry 구현 (의존: Task 1)
- `src/worker-entry.ts`
- `SENA_CONFIG_PATH` 환경변수에서 config 경로 읽기
- dotenv 로드 → config 동적 import → createWorker → start
- 이 파일은 orchestrator가 fork하는 worker 스크립트

### Task 5: CLI 커맨드 구현 (의존: Task 2, 3, 4)
- `src/cli.ts` — 메인 진입점 (#!/usr/bin/env node --import tsx/esm)
- `src/commands/start.ts` — orchestrator 기동 (foreground/-d 데몬)
  - config 로드 → createOrchestrator({ port, workerScript: worker-entry 경로 }) → start
  - 데몬 모드: detached child_process.fork + PID 파일 기록
  - foreground: 직접 실행 + SIGUSR2/SIGINT/SIGTERM 핸들링
- `src/commands/stop.ts` — PID 읽기 → SIGTERM → 대기 → PID 파일 삭제
- `src/commands/restart.ts` — PID 읽기 → SIGUSR2 (worker only) 또는 --full (stop+start)
- `src/commands/status.ts` — PID 확인 + /health HTTP 체크
- `src/commands/logs.ts` — `sena.log` tail (child_process로 tail -f)

### Task 6: 빌드 및 통합 테스트 (의존: Task 5)
- `pnpm build` 전체 통과 확인
- CLI 빈이 올바르게 실행되는지 확인
- 실제 에이전트 디렉토리에서 `sena start` / `sena status` / `sena restart` / `sena stop` 동작 확인

### Task 7: npm 퍼블리시 및 에이전트 마이그레이션 (의존: Task 6)
- `@sena-ai/cli` npm 퍼블리시
- `~/agents/sena_v2/` 마이그레이션: start.ts, worker.ts, restart.sh 제거, package.json에 @sena-ai/cli 추가
- `~/agents/lumie/` 마이그레이션: 동일
- 마이그레이션 후 각 에이전트 정상 기동 확인

## 상세 검증 계획
1. **타입체크**: `pnpm typecheck` 에러 0

6. **로그**: `sena logs` 출력 확인
7. **에이전트 마이그레이션 후 Slack 응답 동작 확인** (sena_v2, lumie)
